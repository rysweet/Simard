# Goal board corruption guards

## The problem this solves

Before issue [#1582](https://github.com/rysweet/Simard/issues/1582) the goal
board could be silently destroyed in a single OODA cycle. The root cause was a
combination of three independent weaknesses:

1. **LLM hallucination in orient.** The prompt-driven orient brain returns a
   list of `Priority` structs. Each `Priority` carries a `goal_id` that is
   generated by the LLM. When the LLM produced a `goal_id` that did not match
   any goal on the active board — for example `"g1"` instead of a real UUID —
   that priority was accepted without validation. Downstream code that indexed
   into the board by `goal_id` could then overwrite or discard real goals.

2. **No integrity check on load.** If a corrupt board had already been
   persisted to disk (from a previous cycle), the next cycle loaded it and
   treated the corrupted data as authoritative. A board containing a single
   goal named `"Goal g1"` would replace the five real goals that existed before
   the corruption event.

3. **No mid-cycle corruption detection.** Nothing in the cycle compared the
   board's state before the core OODA phases began to its state at the end of
   the Curate phase. A goal that silently vanished — without going through
   `archive_completed` — would be persisted as absent.

Together these three gaps could collapse a healthy five-goal board to a single
bogus `g1` entry in one cycle, with no warning in the logs.

---

## How it works now

Three guards operate at different points in the cycle, forming a defence-in-depth:

```
Cycle start
    │
    ├─ Guard 2: board_integrity_suspect()  ← rejects corrupt boards on load
    │
    ├─ Snapshot pre-cycle active IDs       ← baseline for Guard 3
    │
    ├─ Observe → Orient → Decide → Act
    │        │
    │        └─ Guard 1: filter_hallucinated_priorities()  ← per-orient call
    │
    └─ Curate
           │
           ├─ archive_completed()
           │
           └─ Guard 3: corruption check    ← compares post vs pre snapshot
                  │
                  ├─ clean  → persist_board()
                  └─ dirty  → skip persist, preserve disk state
```

### Guard 1 — filter hallucinated priorities (orient phase)

**Where:** `src/ooda_loop/orient.rs`, called from `orient_with_brain` after the
LLM brain returns its `OrientJudgment` priorities and before synthetic goal
priorities are appended.

**What it does:** For each `Priority` in the list returned by the brain, the
function checks whether `priority.goal_id` is either:

- present in the current `active_goals` list, **or**
- a synthetic goal id (starts with `__`, e.g. `__memory__`,
  `__improvement__`, `__eval_watchdog__`).

Any priority that fails both checks is dropped. A warning is written to stderr
for each dropped id:

```
[simard] OODA orient: dropping hallucinated goal_id 'g1' — not on active board
```

Synthetic goals are never sourced from the LLM; the `__`-prefix guard exists as
a safety net for the rare case where a future LLM response happens to produce
that prefix.

**Signature:**

```rust
pub(crate) fn filter_hallucinated_priorities(
    priorities: &mut Vec<Priority>,
    active_goals: &[ActiveGoal],
);
```

### Guard 2 — board integrity check on load

**Where:** `src/ooda_loop/cycle.rs`, immediately after `load_goal_board` returns
a non-empty board at cycle start.

**What it does:** `board_integrity_suspect` inspects every `ActiveGoal` in the
loaded board and returns `Some(reason)` if any goal looks like a placeholder:

| Heuristic | Trigger | Example |
|-----------|---------|---------|
| Short id | `id.len() < 5` | `"g1"`, `"g12"`, `"g123"` |
| Placeholder description | matches `^\s*goal\s*[a-z0-9]{1,4}\s*$` (case-insensitive) | `"Goal g1"`, `"GOAL abc"`, `"goalg1"` |

The 5-character threshold is chosen to catch single-letter + digit hallucinations
while allowing any realistic UUID fragment, slug, or issue number as an id.

When a suspect board is detected the loaded board is silently discarded and the
cycle falls through to `seed_default_board` as if no board existed. A warning is
logged:

```
[simard] OODA start: rejecting loaded board — integrity suspect: goal 'g1' has
suspiciously short id (len 2); falling back to default seed
```

**Signatures:**

```rust
pub(crate) fn board_integrity_suspect(
    board: &GoalBoard,
) -> Option<String>;

pub(crate) fn is_placeholder_description(desc: &str) -> bool;
```

### Guard 3 — pre-cycle snapshot and curate-phase corruption check

**Where:** `src/ooda_loop/cycle.rs`. Snapshot taken just before the Observe
phase; check runs during the Curate phase, just before `persist_board`.

**What it does:** Before the Observe phase begins, the cycle records the set of
active goal ids:

```rust
let pre_cycle_active_ids: HashSet<String> = state.active_goals.active
    .iter()
    .map(|g| g.id.clone())
    .collect();
```

At the end of the Curate phase, after `archive_completed` has run, the cycle
computes which pre-cycle goals are no longer in `active`:

```
vanished = pre_cycle_active_ids
           − post_active_ids          (still active)
           − archived_this_cycle_ids  (legitimately completed)
```

A goal may legitimately leave `active` only by being moved to `archived` via
`archive_completed`. Any goal that is absent from both `active` and the
current-cycle archive is treated as a corruption signal.

If any vanished goals are detected, the cycle **skips `persist_board`
entirely**, leaving the last-known-good board on disk unchanged. An error is
logged:

```
[simard] OODA curate: CORRUPTION DETECTED — 4 goal(s) vanished without archival:
ship-v1-release, improve-memory-persistence, fix-engineer-deadlock,
gym-knowledge-recall; skipping persist to protect board
```

---

## Interaction between the three guards

The guards are complementary, not redundant:

- Guard 2 stops a corrupt board from ever becoming the cycle's working state.
- Guard 1 stops LLM hallucinations from altering the in-memory board during the
  orient phase.
- Guard 3 catches any corruption that slips past Guard 1 (or originates
  elsewhere) and prevents it from reaching disk.

A cycle that starts with a clean board (Guard 2 passed), processes orient
safely (Guard 1 filtered hallucinations), and arrives at Curate with all goals
intact (Guard 3 passes) will persist normally. No code path in a healthy cycle
is affected.

---

## Observability

All three guards emit `eprintln!` lines prefixed with `[simard]`. These appear
in the daemon log (stdout/stderr redirect) under the relevant phase:

| Prefix | Phase | Meaning |
|--------|-------|---------|
| `[simard] OODA orient: dropping hallucinated goal_id ...` | Orient | Guard 1 filtered a priority |
| `[simard] OODA start: rejecting loaded board — integrity suspect: ...` | Load | Guard 2 rejected board |
| `[simard] OODA curate: CORRUPTION DETECTED — ...` | Curate | Guard 3 skipped persist |

Absence of all three prefixes in a cycle's logs means the cycle ran cleanly.

---

## Caveats and known limitations

- **Guard 1 is orient-only.** Other OODA phases (Observe, Decide, Act) are not
  currently validated for hallucinated goal ids. If a future phase directly
  mutates `state.active_goals` based on LLM output, a similar filter should be
  added at that site.

- **Guard 2 relies on heuristics.** A corrupt board with realistic-looking goal
  ids will pass the integrity check. The heuristics were chosen to catch the
  observed failure mode (`"g1"` placeholder ids); they are not exhaustive.

- **Guard 3 does not restore.** When corruption is detected, the cycle skips
  persist but does not attempt to repair or restore `state.active_goals` in
  memory. The next cycle will reload from disk (which is still the last-known-
  good state), so the daemon self-heals without operator intervention.

- **Synthetic goals are unconditionally allowed through Guard 1.** Any
  `goal_id` starting with `__` bypasses the active-board check. This is
  intentional — synthetic goals are appended after the filter runs — but it
  means a future LLM that produces a `__`-prefixed hallucination would not be
  caught.
